# 热点监控与发现系统需求分析

## 1. 项目背景

用户希望第一时间获取指定领域内的热点变化，例如 AI 大模型更新、AI 编程工具动态、重要产品发布、政策变化、融资消息或关键人物发言。

传统做法依赖人工搜索、刷社交媒体、订阅新闻源，存在以下问题：

- 信息来源分散，人工检索成本高。
- 热点传播速度快，人工发现通常滞后。
- 关键词命中不代表内容真实相关，存在蹭热点、假冒、标题党、搬运和误报。
- 用户除了监控明确目标外，还希望系统能主动发现某个主题范围内的新热点。

因此，需要建设一个“自动监控 + 自动发现 + AI 识别 + 实时通知 + 可视化查看 + Agent Skills 能力封装”的热点情报系统。

## 2. 项目目标

### 2.1 核心目标

- 用户输入监控关键词后，系统自动持续监控相关信息源。
- 当关键词相关且经过 AI 判定为真实有效的事件出现时，第一时间发送通知。
- 系统按固定周期自动发现某个主题范围内的新热点，而不是只做关键词匹配。
- 用户可以通过响应式 Web 页面查看热点列表、趋势、来源、可信度和通知状态。
- 系统能力可封装成 Agent Skills，供其他 AI 代理调用，实现“监控、发现、鉴别、通知”的自动化闭环。

### 2.2 业务目标

- 提高用户获取热点的时效性。
- 降低用户主动搜索的频率和成本。
- 降低误报、假冒信息、无关内容带来的噪音。
- 形成可扩展的热点发现基础设施，后续可扩展到更多垂类。

## 3. 用户画像与典型场景

### 3.1 用户画像

- 科技行业从业者：关注 AI 大模型、AI 编程、出海工具、创业公司动态。
- 内容创作者：希望快速发现可写、可评、可转发的热点。
- 投资/研究人员：关注赛道变化、产品发布、融资与关键舆情。
- 高信息敏感用户：希望“比别人更早知道”。

### 3.2 典型场景

#### 场景 A：关键词实时监控

用户输入“Claude 4”“GPT-5.4”“Cursor 新版本”“Gemini CLI”等关键词，当这些关键词对应的真实新动态出现时，系统在几分钟内推送通知。

#### 场景 B：主题范围自动发现

用户输入“AI 编程”作为监控范围，系统无需用户枚举所有关键词，也能定期自动发现该范围内新出现的热点事件、产品更新、社区爆点和高传播内容。

#### 场景 C：真假识别与降噪

当某篇文章标题包含“OpenAI 发布”，但内容只是二手解读、旧闻翻炒或借势营销时，系统应尽量识别并降低优先级，避免误触发高优先级通知。

#### 场景 D：多 Agent 协作

其他 AI Agent 调用该系统技能，发起“监控 AI 大模型更新”“发现过去 6 小时 AI 编程赛道热点”“输出可信热点摘要”等任务，并获得结构化结果。

## 4. 问题定义

该系统需要解决两类不同但相关的问题：

### 4.1 明确目标监控

输入是用户明确指定的关键词、实体、产品名、作者名、公司名、事件名。

系统任务是：

- 追踪新内容是否出现。
- 判断内容是否真实相关。
- 判断是否值得通知。
- 尽快通知用户。

### 4.2 范围型热点发现

输入是用户指定的主题范围，例如“AI 编程”“AI 大模型”“中美科技新闻”“独立开发者工具”。

系统任务是：

- 自动收集该范围内的新内容。
- 识别哪些内容正在形成热点。
- 给出热点排序、聚类和摘要。
- 将新热点以产品化方式展示和通知。

## 5. 产品范围

## 5.1 In Scope

- 用户配置关键词监控。
- 用户配置主题范围监控。
- 多来源信息采集。
- 热点候选内容去重、聚类、排序。
- AI 相关性识别、真假识别、价值判定。
- 多渠道通知。
- 响应式 Web 页面展示。
- Agent Skills 封装。
- 基础订阅管理、通知频率控制、结果回溯。

## 5.2 Out of Scope（第一阶段建议不做）

- 用户社交关系链推荐。
- 全量舆情分析大盘。
- 自建大规模搜索引擎。
- 复杂商业化计费体系。
- 完整移动 App。

## 6. 功能需求

### 6.1 监控配置

用户可新增、编辑、启停以下监控项：

- 关键词监控：例如“OpenAI o4-mini”“Claude Code”“Gemini 2.5 Pro”。
- 实体监控：公司、产品、模型、人物、组织。
- 主题范围监控：例如“AI 编程”“AI Agent”“大模型开源项目”。
- 来源范围配置：新闻站点、社交平台、技术社区、产品发布平台、GitHub、博客、论坛等。
- 语言与地区范围：中文、英文、全球、指定地区。
- 通知优先级：高优先级即时通知、普通优先级汇总通知。

系统应支持：

- 精确匹配、模糊匹配、别名匹配、实体归一化。
- 关键词黑名单与噪音排除词。
- 监控规则版本管理与启停。

### 6.2 数据采集

系统应支持从多个来源定时抓取或轮询热点候选数据。

来源建议包括：

- 新闻媒体与资讯站。
- 社交媒体与社区平台。
- GitHub Release、Issue、Trending、Commit 动态。
- 产品发布平台。
- 官方博客、开发者博客、公司公告页。
- RSS、邮件订阅转采集、公开 API、网页抓取。

采集要求：

- 支持多来源扩展式接入。
- 支持定时采集与高频轮询并存。
- 保存原始内容、抓取时间、来源元信息、正文摘要、链接。
- 支持失败重试、限流、去重和异常监控。

### 6.3 热点候选生成

系统需要从海量原始内容中生成“热点候选事件”。

核心能力包括：

- 标题和正文抽取。
- 实体识别。
- 关键词扩展和同义词扩展。
- 内容去重。
- 近似内容聚类。
- 同一事件多来源归并。

示例：

当“某模型发布新版本”被多个站点同时报道时，系统不应生成多条独立热点，而应聚合为一个事件，并记录多个来源、传播强度和时间线。

### 6.4 AI 相关性识别

这是系统的核心能力之一。用户强调不能只靠关键词命中，必须利用 AI 识别假冒或无关内容。

系统需要对每条候选内容进行多维判定：

- 是否与用户监控项真实相关。
- 是否为旧闻翻炒。
- 是否只是提及但非核心内容。
- 是否存在标题党、蹭热点、借势营销。
- 是否为转载、搬运、二次解读。
- 是否具备足够信息价值。

建议输出以下 AI 判定字段：

- relevance_score：与监控项的相关度。
- authenticity_score：真实性/可信度评分。
- novelty_score：新颖度评分。
- importance_score：重要性评分。
- noise_score：噪音概率。
- reasoning_summary：简短判断理由。
- matched_entities：命中的实体与别名。

### 6.5 假冒内容识别策略

“假冒内容”不仅指伪造新闻，也包括看似相关但实际不构成有效事件的内容。系统应具备以下识别策略：

- 交叉来源验证：是否存在多个独立可信来源佐证。
- 发布时间验证：是否真的是新内容，而非旧文章重新传播。
- 官方源优先：官方网站、官方博客、官方账号、代码仓库发布页权重更高。
- 实体一致性验证：标题中的实体与正文中的实体、链接目标、上下文是否一致。
- 事件真实性验证：是否存在明确动作，例如发布、更新、开源、融资、上线、收购、政策公布。
- 内容类型识别：区分原始新闻、评论、转述、搬运、预测、谣言、广告。

### 6.6 热点发现与排序

对于范围型监控，系统需要主动发现“新热点”，而不是等待用户手动列出关键词。

热点发现可基于以下信号综合判断：

- 短时间内提及量显著上升。
- 多来源同时出现。
- 高影响力来源发布。
- 用户所选范围内实体频繁共现。
- 社区互动快速增长。
- 新事件类型触发，例如版本发布、融资、开源、产品上线、重大政策。

排序维度建议包括：

- 实时热度。
- 增速。
- 来源可信度。
- 与用户范围的贴合度。
- 新颖性。
- 商业/行业影响程度。

### 6.7 通知系统

系统应在高价值事件出现后尽快通知用户。

通知能力包括：

- 即时通知：命中高优先级规则时立即推送。
- 周期汇总：每小时、每 6 小时、每天生成摘要。
- 去重通知：同一事件避免重复骚扰。
- 升级通知：同一事件热度快速上升时再次提醒。
- 通知降噪：低可信或低价值事件不即时推送。

通知渠道建议：

- Web 站内通知。
- 邮件。
- 企业微信/飞书/钉钉机器人。
- Telegram/Discord/Slack。
- Webhook。

通知内容建议包含：

- 热点标题。
- 简短摘要。
- 判定原因。
- 来源数。
- 首次发现时间。
- 热度等级。
- 查看详情链接。

### 6.8 响应式 Web 页面

用户要求以产品形式呈现，且需兼容响应式 Web 页面。

页面建议至少包括以下模块：

- 首页/控制台：展示今日热点、最新发现、关注范围摘要。
- 监控管理页：管理关键词、范围、来源、通知策略。
- 热点流页面：按时间倒序展示最新热点。
- 热点详情页：展示摘要、来源聚合、AI 判断、传播时间线。
- 趋势页：展示某主题下热点演化、热度变化、来源分布。
- 通知记录页：查看历史推送、是否送达、用户是否点击。

响应式要求：

- 桌面端适合高密度信息浏览。
- 移动端适合快速查看热点、清晰触达通知入口。
- 关键指标在小屏下保持可读性。

### 6.9 Agent Skills 封装

系统需封装成可供其他 AI 调用的 Agent Skills，核心不是页面展示，而是将能力抽象为标准化技能接口。

建议封装以下技能：

- monitor_keywords：监控指定关键词并返回命中结果。
- discover_trends：发现指定主题范围内的最新热点。
- validate_event：校验某条候选内容是否真实、相关、值得关注。
- summarize_hotspots：将多个热点汇总为结构化摘要。
- push_alert：根据规则发送通知。
- explain_trend：解释某个热点为何值得关注。

技能输入示例：

- 监控范围。
- 时间窗口。
- 优先级阈值。
- 来源约束。
- 输出格式要求。

技能输出建议为结构化 JSON，便于其他 Agent 继续编排。

## 7. 业务流程

### 7.1 关键词监控流程

1. 用户配置关键词/实体与通知规则。
2. 系统持续采集多来源内容。
3. 生成候选事件并完成实体归一。
4. AI 判定内容是否真实相关。
5. 若达到通知阈值，则立即推送。
6. 结果写入 Web 页面和历史记录。

### 7.2 主题范围发现流程

1. 用户配置主题范围与来源范围。
2. 系统周期性抓取该范围内内容。
3. 对内容做聚类、去重、热度计算。
4. AI 识别哪些事件构成真正热点。
5. 按热度、新颖度、可信度排序。
6. 生成热点列表、摘要与推送内容。

## 8. 角色与权限

第一阶段可简化为单用户或轻量多用户模型。

建议角色：

- 普通用户：配置监控项、查看热点、接收通知。
- 管理员：管理数据源、系统参数、模型阈值、通知通道。
- 外部 Agent：通过 Skill/API 调用监控与发现能力。

## 9. 非功能需求

### 9.1 时效性

- 高优先级监控事件需要接近实时发现。
- 对快节奏来源可支持分钟级轮询。
- 通知链路应尽量缩短处理延迟。

### 9.2 准确性

- 尽可能降低误报率。
- 同时避免过度保守导致漏报。
- 支持持续优化 AI 判定阈值。

### 9.3 稳定性

- 数据抓取失败可重试。
- 单一来源异常不影响整体能力。
- 通知失败需重试并保留状态。

### 9.4 可扩展性

- 新来源应可插件化接入。
- 新主题与新垂类可复用现有管线。
- Agent Skills 需可扩展更多任务接口。

### 9.5 可解释性

- 用户需要知道系统为什么判定为热点。
- 用户需要知道为什么触发通知或未触发通知。
- AI 结论应附带简短理由和证据来源。

## 10. 数据模型建议

核心实体建议包括：

- User：用户信息。
- MonitorRule：监控规则。
- SourceRecord：原始来源记录。
- EventCandidate：候选事件。
- HotspotEvent：聚合后的热点事件。
- EventSourceLink：热点与来源映射。
- AIEvaluation：AI 评估结果。
- NotificationTask：通知任务。
- NotificationLog：通知日志。
- AgentInvocationLog：技能调用日志。

## 11. 关键算法与判定思路

### 11.1 相关性判定

使用关键词匹配 + 实体识别 + 语义理解联合判断，避免“仅标题命中”。

### 11.2 新颖度判定

将候选内容与历史事件库比对，识别是否为旧闻重复传播。

### 11.3 事件聚类

将同一事件在不同来源、不同语言、不同表达方式下的内容聚合成一个热点对象。

### 11.4 热度计算

结合提及量、来源权重、时间衰减、互动强度、增速等因素计算。

### 11.5 通知阈值策略

将真实性、相关性、热度、重要性结合成通知分数，分为即时通知、汇总通知和仅记录不推送。

## 12. MVP 建议

为了降低第一阶段复杂度，建议 MVP 只覆盖最能体现价值的能力。

### 12.1 MVP 功能

- 用户配置关键词监控。
- 用户配置少量主题范围监控，例如“AI 大模型”“AI 编程”。
- 接入有限但高价值的数据源。
- 基于 AI 的相关性与真实性判定。
- 邮件/Webhook/站内通知三选二。
- 响应式热点列表页与详情页。
- Agent Skills 提供基础的 monitor 和 discover 两类能力。

### 12.2 MVP 目标

- 先证明“比人工更快发现热点”。
- 先证明“误报控制可接受”。
- 先证明“发现型热点流有持续可看性”。

## 13. 成功指标

建议关注以下指标：

- 热点发现时延。
- 高价值热点命中率。
- 误报率。
- 用户点击通知率。
- 用户保留监控规则数。
- 热点详情页访问频率。
- Agent Skills 调用成功率与复用率。

## 14. 风险与挑战

- 信息源质量参差不齐，容易引入噪音。
- 单纯依赖关键词会带来大量误报。
- 过度依赖 AI 判断可能带来成本与延迟压力。
- 多来源聚类与事件归并难度较高。
- 通知过多会造成用户疲劳，需要强去重与优先级管理。

## 15. 后续演进建议

### 15.1 产品演进

- 引入个性化偏好学习。
- 引入热点订阅模板，例如“AI 大模型更新模板”。
- 增加趋势周报、专题情报页、热点时间线。

### 15.2 技术演进

- 构建来源接入插件体系。
- 构建事件知识图谱与实体库。
- 引入反馈学习机制，用用户行为反哺 AI 判定。
- 对 Agent Skills 开放 API 化与工作流编排能力。

## 16. 结论

该项目本质上不是简单的“关键词订阅器”，而是一个面向热点情报的自动化发现系统。它需要同时解决“监控指定对象”和“自动发现新热点”两个问题，并在此基础上加入 AI 真伪识别、事件聚合、优先级通知和可视化展示。

从产品价值上看，最核心的差异化能力有三点：

- 发现得快。
- 判得准。
- 用得起来。

因此，需求设计必须围绕“时效性、准确性、产品化交付”三个维度展开，优先做出一个能够真实帮助用户抢先发现热点的 MVP，再逐步扩展为可复用的热点监控基础设施与 Agent Skills 能力中心。
