# 热点监控系统技术方案（V1）

## 1. 目标与约束

### 1.1 目标

- 先交付可用 Web 版：支持监控规则配置、热点发现、AI 鉴别、通知、趋势查看。
- Web 功能验收后，再交付 Agent Skills。
- 接入 OpenRouter 作为 AI 服务，完成“相关性 + 真实性 + 新颖度 + 噪音”评估。
- 前端采用“剑来动漫主题”，并满足桌面与移动端响应式。

### 1.2 约束

- 第一阶段优先可运行与可验证，不追求全量数据源。
- 需要保留无 OpenRouter Key 的降级运行能力。
- 通知先支持站内通知 + Webhook，企业 IM 渠道作为后续扩展。

## 2. 总体架构

采用单体分层架构，先降低复杂度，后续可拆分服务。

1. 展示层（Web 前端）

- 规则管理、热点流、通知记录、趋势看板。

2. 接口层（REST API）

- 提供规则、扫描、热点、通知、趋势、技能校验接口。

3. 领域层（监控与评估）

- 候选抓取、去重、事件构建、AI 评估、通知决策。

4. 基础设施层

- 数据源连接器（RSS/API）、OpenRouter 客户端、通知客户端、存储。

## 3. 技术选型

### 3.1 后端

- Node.js + Express
- 原因：实现速度快、生态成熟、易于后续增加定时任务、AI 调用、Webhook 与技能入口。
- 说明：V1 先采用单体服务，后续如果数据源与通知量增长，再拆分采集服务、评估服务和通知服务。

### 3.2 前端

- React + Vite + TypeScript
- 原因：
  - 组件化更适合规则面板、热点卡片、趋势图、通知列表等复杂页面。
  - Vite 启动快、调试轻、适合 V1 快速迭代。
  - TypeScript 可以减少规则字段、事件结构、AI 返回结构上的类型偏差。
- 样式方案：原生 CSS Variables + 组件级样式，保留“剑来”视觉主题的一致性。
- 备选方案：如果需要进一步压缩首版工作量，可临时用纯 HTML + CSS + JS，但正式推荐仍是 React 方案。

### 3.3 数据与任务

- SQLite + Prisma
- 原因：
  - SQLite 适合 V1 本地快速持久化，部署和开发门槛低。
  - Prisma 便于定义规则、热点、通知、扫描日志等模型，并支持后续无痛迁移到 PostgreSQL。
- 任务方式：
  - V1 使用进程内定时扫描。
  - 后续如需稳定调度与并发控制，再引入队列或独立 worker。

### 3.4 AI 服务

- OpenRouter Chat Completions
- 模型默认建议：`openai/gpt-4o-mini`（可配置）
- 输出格式：强制 JSON 结构，便于后续规则判断、告警与页面展示。
- 降级策略：当 OpenRouter 不可用时，回落到本地启发式评估，保证主链路可运行。

### 3.5 通知

- 站内通知：Socket.IO 实时刷新
- 外部通知：Webhook（飞书、钉钉、企业微信、Slack 等均可通过 webhook 接入）
- 说明：
  - Socket.IO 负责前端实时更新，适合热点发现后的即时刷新。
  - Webhook 负责对接外部 IM 和自动化平台。
  - 邮件能力如果后续需要，可作为 Webhook 的扩展通道单独加入。

### 3.6 数据源接入

- V1 优先 RSS、公开 API、官方博客、Hacker News、GitHub Release 等高质量源。
- 后续扩展网页抓取与交互式采集。
- 原则：先保证可信度，再扩来源覆盖面。

## 4. 模块设计

## 4.1 配置模块

- 读取环境变量：端口、OpenRouter Key、模型名、扫描间隔、Webhook URL。
- 配置缺失时给出可运行默认值。

## 4.2 规则管理模块

- 支持 `keyword` 与 `topic` 两类规则。
- 字段：`id/name/type/query/priority/enabled/createdAt/updatedAt`。

## 4.3 数据采集模块

- V1 接入高质量 RSS 源（官方博客 + 技术社区）。
- 输出统一结构：`title/summary/link/source/publishedAt`。
- 要求：失败隔离、超时控制、单源异常不影响整体扫描。

## 4.4 候选处理模块

- 去重：按 `ruleId + link` 去重。
- 预过滤：关键词规则先做快速词匹配，范围规则直接进入评估。
- 事件生成：生成热点事件 ID、打分、判定理由。

## 4.5 AI 评估模块（OpenRouter）

### 输入

- 规则信息（类型、查询、优先级）
- 候选内容（标题、摘要、来源、链接、时间）

### 输出（标准字段）

- `relevance_score`
- `authenticity_score`
- `novelty_score`
- `importance_score`
- `noise_score`
- `reasoning_summary`
- `matched_entities`

### 降级策略

- 无 Key / 请求失败时，采用启发式评估返回同结构字段，确保主流程不中断。

## 4.6 通知决策模块

- 综合评分公式：
  - $score = 0.35R + 0.25A + 0.15N + 0.25I - 0.20Noise$
- 阈值策略：
  - 高优先级规则：`score >= 0.62` 即时通知
  - 普通规则：`score >= 0.70` 即时通知
  - 低于阈值仅入库展示，不推送

## 4.7 前端展示模块（剑来主题）

- 视觉方向：古风卷轴 + 刀剑金属质感 + 水墨雾层背景。
- 页面模块：
  - 立规（规则创建）
  - 剑谱（规则列表）
  - 热榜流（热点事件）
  - 风云势（趋势聚合）
  - 飞剑传书（通知记录）
- 响应式要求：
  - 桌面双栏/多栏
  - 移动单栏
  - 卡片信息不溢出、按钮触达可用

## 4.8 Agent Skills 模块（第二阶段）

- 技能清单：
  - `monitor_keywords`
  - `discover_trends`
  - `validate_event`
- 形式：manifest + CLI + 后端技能接口。
- 先依赖已稳定的 Web API，避免重复实现业务逻辑。

## 5. API 设计（V1）

### 5.1 系统

- `GET /api/health`：服务状态与 AI 接入状态。

### 5.2 规则

- `GET /api/rules`：规则列表。
- `POST /api/rules`：新增规则。
- `PATCH /api/rules/:id`：启停/修改规则。

### 5.3 扫描与结果

- `POST /api/scan`：立即执行一次扫描。
- `GET /api/hotspots`：热点事件列表。
- `GET /api/trends`：趋势聚合结果。
- `GET /api/notifications`：通知记录。

### 5.4 技能接口（第二阶段）

- `POST /api/skills/validate`：给定候选内容与 query，返回 AI 评估结构。

## 6. 数据结构设计

## 6.1 Rule

- `id, name, type, query, priority, enabled, createdAt, updatedAt`

## 6.2 HotspotEvent

- `id, ruleId, ruleName, title, summary, link, source, publishedAt, scannedAt`
- `ai`（评估对象）
- `score`

## 6.3 Notification

- `id, title, message, score, source, link, createdAt, reason`

## 6.4 ScanResult

- `scanAt, candidates, activeRules, newEvents`

## 9. 风险与应对

1. 数据源质量不一

- 应对：白名单源优先 + 可信源加权。

2. OpenRouter 波动或限流

- 应对：降级策略 + 重试 + 超时熔断。

3. 通知骚扰

- 应对：去重、阈值、升级通知策略。

4. 重启丢数据（V1 内存存储）

- 应对：V1.1 引入 SQLite 持久化。
